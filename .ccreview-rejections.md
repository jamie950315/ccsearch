## Over-aggressive fallback for < 50 chars
- **Suggestion**: Require at least one structural SPA signal before fallback when content is < 50 chars, instead of unconditionally triggering.
- **Rejection reason**: After Content-Type (HTML only) and status_code (200) guards, only valid HTML pages reach SPA detection. An HTML page with < 50 chars of visible text is almost certainly a SPA skeleton or blocked page. The fallback is non-destructive (keeps original if FlareSolverr isn't better), so cost is only latency for rare edge cases.
- **Rejected**: 2 times.

## Mount-point matching too brittle
- **Suggestion**: Use BeautifulSoup/lxml DOM parsing to match SPA mount point IDs instead of string matching, to handle single quotes, extra whitespace, and attribute reordering.
- **Rejection reason**: Current approach uses lowercase string matching which handles case variations. Real-world HTML almost universally uses double quotes. Adding a second BeautifulSoup parse pass for rare single-quote variants is over-engineering.
- **Rejected**: 1 time.

## Length-only "better result" heuristic
- **Suggestion**: Use a quality heuristic (paragraph count, boilerplate ratio, title presence) instead of raw text length to decide if FlareSolverr result is better.
- **Rejection reason**: Adds significant complexity for marginal benefit. Length comparison works well for the primary use case (SPA empty shell vs rendered content). Can revisit if real-world false negatives emerge.
- **Rejected**: 1 time.
